# Repository Guidelines

## Project Structure & Module Organization

This repository contains a VS Code extension for remembering, restoring, and converting file encodings.

- `src/extension.ts` contains the extension activation code, command handlers, encoding record storage, status bar updates, and conversion logic.
- `src/test/extension.test.ts` contains Mocha-based extension tests.
- `assets/` contains marketplace and extension icons.
- `dist/` is generated by esbuild and is the extension runtime entry point.
- `.github/workflows/` contains packaging and publishing workflows.
- `sandbox/` is for local manual test data, not production source.

## Build, Test, and Development Commands

- `npm install` installs dependencies.
- `npm run compile` runs type checks, ESLint, and bundles to `dist/extension.js`.
- `npm run watch` runs TypeScript and esbuild watchers for local extension development.
- `npm test` compiles tests, compiles the extension, lints source, and runs VS Code extension tests.
- `npm run package` creates a production bundle.
- `npm run vsix` builds a local `.vsix` package with `vsce`.

Use `F5` in VS Code to launch an Extension Development Host.

## Coding Style & Naming Conventions

Use TypeScript with strict checking enabled. Follow `src/extension.ts`: tabs for indentation, single quotes, explicit helper functions, and `async`/`await` for VS Code APIs. Command IDs should use `encodingKeeper.*`, for example `encodingKeeper.convertGbk`. Keep encoding labels user-facing and encoding values compatible with VS Code or `iconv-lite`.

Run `npm run compile` before submitting changes; it includes linting and bundling.

## Testing Guidelines

Tests use Mocha with VS Code's extension test runner. Add tests under `src/test/` and name files `*.test.ts`. Prefer behavior-focused tests for command registration, record-file handling, and encoding conversion edge cases. For manual checks, test file and folder context menu flows, including dirty files and mixed encodings.

## Commit & Pull Request Guidelines

Recent commits are short and imperative, such as `add Transcoding Function` or `switch 0.0.6`. Keep messages focused on one change.

Pull requests should include a short description, test results such as `npm run compile` or `npm test`, and notes for user-visible menu or command changes. Link issues when available. Include screenshots only when UI text, menus, or Marketplace assets change.

## Security & Configuration Tips

Do not commit local workspace encoding records from test projects unless intentional. Avoid committing generated `.vsix` files, `node_modules/`, or temporary sandbox output. When changing encoding conversion behavior, be careful with destructive writes and preserve the existing dirty-file safeguards.
